A few months back, I was asked to put together a short demonstration on how I would automate IP Address Management (IPAM) in Microsoft Azure as new environments are spun up. If you are hybrid multi-cloud, it probably means you are sharing the same RFC1918 space across clouds and traditional networking. If this is the case, it means you probably want to keep track of allocation across VNets, Subnets, and VMs. The requirements included:

  • Minimal human intervention.
  • Repeatable across other public cloud providers.
  • Push button automation demonstrated in real time.
  • Must be compatible across traditional and cloud networking.

Why automate here?

If Tesla can make self driving cars and my coffee maker turns itself on in the morning, we can automate IPAM across the hybrid multi-cloud network right? Also, cloud demands agility. If you want agility, you need design patterns to build something repeatable against. If you are managing IPAM in the cloud with any Microsoft Office tool, then you are doing it wrong.

Getting Started

In keeping things simple, I tried to focus on the foundational elements only for the context of this demo. In thinking of standing up a new environment from scratch, let’s create a checklist of what we intend to build and identify how it all fits together logically.

  1. Resource Group to hold logical objects.
  2. VNet to establish the private network.
  3. NSG with a standard policy (sane defaults).
  4. A few Subnets to demonstrate segmentation.
  5. Modify prefix-list on physical gear.

Why Ansible and Netbox?

Leveraging something like Ansible begins to really make sense when the goal is automation at scale for networking across multiple vendors and environments. Netbox can serve as a Source of Truth intended to represent the desired state of a network versus its operational state. The API is very flexible, and the functionality can drive many automation use cases well beyond IPAM.

Prerequisites

This post will not detail how to deploy Ansible or Netbox as there is plenty of examples out there already. There is great documentation for setting up both of these platforms via Docker. Instructions for setting up Ansible AWX can be found here and Netbox here.

To authenticate to an Azure subscription, you will need to create a Service Principal. Once a Service Principal is created, you will need to assign a role so that you can access the resources in that subscription. Detailed instructions for completing these steps can be found here.

Design Drives Automation

Thinking through a given design is a key element for how you approach the automation. In cloud, you generally have a hierarchy in which logical components exist which may contain additional logical components.

With Azure for instance, management groups contain subscriptions. Subscriptions contain Resource Groups which contain VNets. A VNet may be a shared resource across many apps and services in scope across many teams. For this demo, let’s take this into consideration and leverage two different standard workflows:

  • Base Workflow Builds Foundational components that can be shared across business units, teams, applications, and tools.
  • Service Workflow Builds Service oriented components specific to a given team, service, and application environment.

Following The Logic

A great benefit of using Ansible is the provided flexibility. This can also cause major confusion as there are many ways in which you can structure things. It is beneficial to understand logic flow and inheritance. As an example, we can follow the logic for reserving a subnet in Netbox.

High Level Plays

The playbooks sitting in our root project folder are pretty simple. They are primarily used as the entry for Ansible and execute based on a specific condition. This was my first time passing through logic in this manner from Ansible Tower, and it worked well for this use case.

play.reserve_azure_snet.yml

Functional Roles

Roles for this demo are organized by either a given platform or OS type. Each role’s main file will include specific tasks to use based on the condition defined in the root level play.

main.yml

Purpose Built Tasks

The actual logic is handled inside a given role’s tasks. Each task is purpose based and does a very specific thing. Also, the idea is to keep things DRY (Don’t repeat yourself) so that we can easily reuse things. This means we don’t want to statically set values here. The smaller a thing is, the less it does. If something does less, it makes it easier to repeat.

task.reserve_azure_snet.yml

The Physical Network

Out of the three platforms we have interacted with here, the Cisco ASR is the only one that isn’t API driven. For automating this beauty, I’ll be using Ansible network_cli for communication and ios_config for configuration.

There are more modern ways to do this today but in practice this is probably reality for most. This is an extremely simple use case, so this type of execution serves its purpose for the scope of a demo. Each time the Service Workflow is run, the subnet it reserves will be appended to a prefix-list living on the ASR.

In a real world cloud scenario, this prefix-list could be used to identify and filter traffic. It could then be used inside a route-map to enable and enforce policy criteria beyond the routing table. When doing BGP to cloud, this can be important as you may want to enforce specific policy on specific neighbors to specific clouds.

task.append_prefix_list.yml

Putting it together with Ansible Tower

Ansible Tower offers two flavors of templates. Job Templates are used to execute a single task many times, while Workflow Templates stitch together multiple Job Templates.

The value here aims at getting the same outcome that your typical Build/Release pipelines would offer for Infrastructure as Code. When thinking of infrastructure as Code I generally think immutability and cloud exclusivity.

In my opinion, this approach lends itself more to remaining flexible and accommodating for a multitude of disparate and traditional infrastructure while also providing the ability to integrate with new technologies.

Job Templates

In the spirit of building for repeatability, I created Job Templates for each unique task. This allows me to reuse them over time in new Workflow Templates as new use cases present themselves.

Workflow Templates

I then assembled my two Workflow Templates that I described above. The Base Workflow in practice would be run far less frequently than the Service Workflow.

During this demo, I ran the Base Workflow once to create my foundational components and the Service Workflow three times to deploy a hypothetical app, db, and web tier environment.

Job Surveys

Surveys are a great way to interactively populate variables at runtime. Once a source of truth in a given data domain is populated and orchestration logic is mature, this is something you will probably use less.

Executing the Workflows

I’m not a huge fan of Web UIs, but running a demo like this in which a large part of your audience is non-technical can be very useful. It gives you just enough and not too much. As the workflow runs, it uses the universal green-go-complete and red-stop-failed during progress and through to completion.

Netbox Prefix Validation

In looking at Netbox, we see that our VNet and Subnets now exist with the correct role, enforced tags, and description.

Azure Resource Validation

Azure is now showing our new Resource Group, VNet, and NSG. In a production grade environment, it would make sense to build a cohesive tagging strategy across cloud providers and into a tool like Netbox.

This enables you to truly reference an environment in its entirety even through the traditional network. The desired outcome would be automating the network from end-to-end both with initial deployment and decommission.

Azure Prefix Assignment

If we dig down further into our VNet configuration, we can see the prefix matches what was reserved in Netbox. We can then validate each of the subnets have the correct prefix and the the base NSG is attached to each of them.

Cisco ASR Prefix-List Update

In an effort to bridge the cloud and traditional network automation into a single Workflow, I thought this was a practical use-case. It is not a good practice to just advertise all prefixes in and out of each cloud as new environments are spun up.

Conclusion

All I have to say is, building out a logical way for developers moving at the speed of cloud to consume IPAM responsibly is WINNING! Now, excuse me while I go dance in the rain…
